Файлы:
- `yadisk_sync_common.py`
- `yadisk_upload.py`
- `yadisk_download.py`
- `sync.sh`

Запуск:
    python yadisk_upload.py --local "/path/to/local" --remote-root "disk:/backup" --token "TOKEN"
    python yadisk_download.py --local "/path/to/local" --remote-root "disk:/backup" --token "TOKEN"

Рекомендуемый запуск для `./runs/`:
    ./yadisk/sync.sh upload OPTIONAL_SUBDIR
    ./yadisk/sync.sh download OPTIONAL_SUBDIR

Что считается объектом:
- обычный файл, который не лежит внутри директории-объекта, это отдельная независимая задача;
- если директория напрямую содержит `plots/`, то объектом считается вся эта директория целиком;
- если на Яндекс Диске директория напрямую содержит `plots.zip`, то объектом считается вся эта директория целиком;
- один worker берёт такой объект целиком и обрабатывает весь его subtree последовательно;
- между разными объектами и отдельными файлами используется общий pool воркеров.

Правила сравнения:
- обычные файлы считаются совпавшими, если совпадает md5;
- если удалённый md5 недоступен, файл считается несовпавшим;

Правила для `plots`:
- локальная папка `plots/` загружается как `plots.zip`;
- на Яндекс Диске любой `*.zip` хранится как `*.zipfast`;
- upload для директории-объекта сначала синхронизирует всё кроме `plots`, а потом:
  - если весь остальной subtree уже точно совпадает и удалённый `plots.zip` существует, `plots` считается синхронизированным без локальной сборки zip;
  - иначе `plots/` архивируется, сравнивается с удалённым `plots.zip` по md5 и при необходимости загружается;
- download для директории-объекта сначала синхронизирует всё кроме `plots`, а потом:
  - если по ходу этого прохода не понадобилось менять ничего кроме `plots`, и локальный `plots/` уже существует, `plots` считается синхронизированным;
  - иначе скачивается `plots.zip` и распаковывается в `plots/`.

Что скрипты делают и чего не делают:
- лишние файлы ни локально, ни удалённо не удаляются;
- дополнительные fallback-ветки не используются: если происходит неожиданная ошибка, скрипт падает с traceback;
- retry есть только у upload/download: любая неуспешная попытка логируется и повторяется;
- временные zip-файлы создаются в `./yadisk/tmp/`.

Логирование:
- формат: `[current/total] ACTION ...`;
- одна строка на обычный файл;
- одна строка на каждый `plots`-объект;
- дополнительные строки появляются только для сетевых retry.
